# Contributing Workflow

## Purpose

Документ описывает единый workflow выполнения задач в проекте MAX Java Bot Framework.

## 1. Как брать задачу

1. Зафиксировать формулировку задачи и ожидаемый результат.
2. Определить, это:
- кодовая задача;
- архитектурная задача;
- документационная задача.
3. Проверить влияние на существующие контракты (`docs/*.md`, `docs/adr/*.md`, public API).

## 2. Как планировать изменения

Перед реализацией нужно:

1. Кратко описать, что будет изменено и почему.
2. Перечислить затрагиваемые модули/файлы.
3. Проверить, нет ли конфликта с уже принятыми ADR.
4. Если задача большая, разбить на минимально целостные подзадачи и коммиты.

## 3. Как выполнять изменения

1. Сначала править контракты/документацию, если меняется поведение API.
2. Затем вносить кодовые изменения в соответствующих модулях.
3. Не смешивать несвязанные изменения в одном коммите.
4. Держать границы client/runtime согласно ADR-0003.

## 4. Обязательное обновление README

README обновляется после каждой завершённой задачи, если изменилось хотя бы одно из:
- текущий статус проекта;
- список/ссылки на документацию;
- модульность/архитектурные решения;
- публичные контракты API;
- команды запуска/сборки/тестов.

Если README не требует правки, это должно быть явно проверено перед коммитом.

## 5. Проверка перед commit

Минимальный чек-лист:

1. Просмотреть `git diff` и убрать мусорные изменения.
2. Запустить релевантные тесты/сборку.
3. Убедиться, что изменения не ломают заявленные public contracts.
4. Обновить документацию (`README` + профильные docs) при необходимости.

## 5.1 Тестовые JSON fixtures (client SDK)

Для `max-client-core` используем единый подход:

1. Добавлять sample ответы MAX API в `max-client-core/src/test/resources/fixtures`.
2. В тестах использовать helper `ru.tardyon.botframework.client.test.JsonFixtures`:
- `JsonFixtures.read("name.json")` для serialization tests;
- `JsonFixtures.jsonResponse("name.json")` для `MockWebServer` transport tests.
3. Для client transport tests использовать `ru.tardyon.botframework.client.test.MockHttpClientTestContext`, чтобы не дублировать:
- `MockWebServer` lifecycle;
- client/config/transport bootstrap;
- utility-методы `enqueue/takeRequest`.
4. Не дублировать один и тот же JSON inline в нескольких тестах, если это типовой ответ API.

## 6. Когда делать commit

Commit делается только когда:

1. Задача завершена как минимально целостный инкремент.
2. Проверки выполнены настолько, насколько позволяет среда.
3. README и docs синхронизированы с изменениями.

## 7. Формат commit message

Предпочтительный стиль — conventional/near-conventional:

- `feat(dispatcher): add router inclusion API`
- `fix(client): handle MAX API 429 response`
- `docs(readme): document callback handling`
- `refactor(core): extract update processing pipeline`
- `test(router): add first-match routing coverage`

Правила:
- scope должен отражать модуль или слой;
- сообщение должно описывать, что изменилось, а не общий контекст;
- один commit = одна логическая цель.

## 8. Если нельзя прогнать тесты/сборку

Если среда не позволяет выполнить проверки:

1. Явно зафиксировать, что именно не удалось запустить и почему.
2. Всё равно завершить задачу с максимально корректной документацией.
3. Привести точные команды для локального запуска, например:

```bash
./gradlew clean test
./gradlew clean build
./gradlew :max-dispatcher:test
```

Нельзя скрывать факт, что проверки не были выполнены.

## 9. Если нельзя сделать git commit

Если commit технически невозможен в среде:

1. Явно указать, что commit не выполнен.
2. Подготовить точные команды для ручного выполнения:

```bash
git add -A
git commit -m "<message>"
```

3. Приложить рекомендуемый commit message.

Нельзя утверждать, что commit сделан, если это не так.

## 10. Definition of Done

Задача считается завершённой, когда:

1. Изменения реализованы и согласованы с архитектурными контрактами.
2. README и профильные docs актуальны.
3. Проверки выполнены или честно зафиксированы ограничения среды.
4. Сделан атомарный commit (или подготовлены точные команды, если commit невозможен).
